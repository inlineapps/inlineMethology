Lyft — marketplace experimentation with time/region splits + adaptive bandits 2016

Lyft's published approach is the close cousin of DoorDash's switchback: every product change runs as an experiment, but marketplace interference forces designs richer than ordinary A/B — time-split and region-split tests.

Canonical source: Nicholas Chamandy's "Experimentation in a Ridesharing Marketplace" (Lyft Engineering, September 2, 2016). Later Lyft posts (~2022) document growing use of contextual bandits for adaptive, always-on optimisation.

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.
🎓 8th-grade version
Lyft uses three kinds of experiments depending on what they're testing. (1) For simple things like a button color, they do a normal A/B test — half see new, half see old. (2) For anything that touches the pool of drivers (like a new pricing rule), they can't use A/B because the two groups affect each other through the shared driver pool — so they switch the whole city on for 30 minutes, off for 30 minutes, on again, and compare. (3) For decisions that happen non-stop (which notification to send, which homepage to show), they use a bandit — an algorithm that keeps trying different options and sends more traffic to the ones that work best. The rule: the design must match the kind of decision, or your number is lying to you.

The honest headline: A/B for diffs, time/region splits for marketplace, bandits for "no off-switch" optimisation

Lyft's engineering blog frames experimentation as three design choices that map to three kinds of decisions. Most teams pick one design and apply it everywhere; Lyft argues the design has to match what's being changed. The decision rule (OEC + measured causal effect) is constant; the shape of the test isn't.

The most under-talked part of Lyft's story: contextual bandits — instead of a discrete A/B that ends, the algorithm learns the best variant per context continuously. It's the design for problems that never stop (matching, dispatch, surge) where "ship the winner" isn't the goal — "always serve the right one" is.

Three experimental designs Lyft uses, and when

The pattern: match the design to the interference structure of the feature. Skipping this step is the most common cause of biased marketplace results.

A/B

User-level A/B — for non-marketplace changes

Standard split. Used for UI, copy, recommendations, anything that doesn't change a shared resource (Dasher pool / driver supply / pricing). The default and the cheapest design.

Time-split

Time-split / region-split — for marketplace changes

For features that touch the supply pool, randomise time windows (or whole regions) to treatment / control. Avoids the interference Lyft repeatedly publishes about; same family as DoorDash switchback.

Bandits

Contextual bandits — for always-on optimisation

For decisions made continuously (matching, surge, ETA), the test never ends. Bandits route more traffic to better-performing variants over time, balancing exploration and exploitation. Lyft's published bandit work is more recent than the 2016 origin post — documented in later engineering posts (~2022) and an academic paper on real-time reinforcement learning for driver supply.

Sources: Nicholas Chamandy — "Experimentation in a Ridesharing Marketplace" (Lyft Engineering, September 2, 2016) — the canonical post · John Kirn — "Challenges in Experimentation" follow-up · later bandit work covered in InfoQ's 2022 summary of Lyft's experiments-beyond-A/B.

How Lyft actually picks the right design

1

Classify the change first

Does it touch a shared resource? Is it a one-time decision or continuous? The answer routes the design. Skipping this is where most marketplace experiments go wrong.

2

If shared resource → time/region split

Randomise the whole city for some time windows. Use cluster-robust standard errors when analysing.

3

If continuous-decision → bandit

Define the reward signal and the contexts. The bandit allocates traffic over time; you read aggregate performance, not a one-shot lift.

4

Otherwise → ordinary A/B

The simple case. Same OEC + guardrails discipline as every other platform on this list.

5

Read the OEC. Decide. Iterate.

The decision rule is unchanged across designs — but the estimate is only trustworthy if the design was right for the change.

Apply to a feature sheet

Lyft's ledger has a design column that's set before any other: A/B for ordinary user-level changes, switchback for shared-resource marketplace features, bandit for continuous always-on decisions. The wrong design produces a biased lift — picking the right one is more important than picking the right metric.

Try it Monday morning (30 minutes). Open your team's backlog. Add a design column. For each item, write A/B / switchback / bandit. Three questions to route: (1) does it touch a shared resource? → switchback. (2) is the right answer context-dependent and the decision never naturally ends? → bandit. (3) otherwise → A/B. Tag every backlog row. Now look at how many items in your backlog have ever been tested with the right design — that's the gap Lyft's first-design-then-feature rule closes.
Quick glossary for the columns below. Time-split / switchback = the whole region runs treatment for some windows, control for others; comparing windows estimates the marketplace-corrected effect. Contextual bandit = an adaptive algorithm that allocates more traffic to better-performing variants while still exploring; "the winner" is allowed to be context-dependent. Reward signal = the OEC for a bandit. Exploration vs exploitation = the tradeoff a bandit balances — keep trying variants we might be wrong about (exploration) vs send traffic to the current winner (exploitation).

The most important reading skill on this page. Notice the Adopt — leave on verdict for the homepage-hero and notification-template bandit rows. Those aren't normal "ship the winner and turn the test off" decisions — there is no single winner, because the right homepage/template depends on user context (time of day, location, history). The bandit keeps running forever, learning per-context. A team's instinct is to "ship the winning variant and move on" — but for these problems, doing so throws away most of the lift. The bandit's value isn't a one-time number; it's the ongoing routing decision.
Decision rule. The design column is the first gate — anything touching shared resources (dispatch, surge, pool, allocation) cannot be a user-level A/B without biased estimates. Bandits are the right shape when "the winner" depends on context (which hero, which template) and the test never naturally ends. Standard A/B is fine for everything else. The aggressive-surge row shows the marketplace kill — trips up but satisfaction down means the metric you optimised wasn't the metric you cared about.
The transferable insight: bandits aren't only for ML teams. A small adaptive system that "tries variants and routes more traffic to the better one" is buildable in a sprint for problems like which homepage hero to show, which merchant to feature, which notification template performs. For decisions that don't have a single right answer over time, bandits beat fixed A/B because they keep learning.
