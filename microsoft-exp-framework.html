Microsoft ExP — OEC, CUPED, and trustworthy experiments at scale

Pioneered by Ron Kohavi and team. Three pieces — a platform that runs experiments cheaply, a single agreed success metric per test (OEC), and a variance-reduction technique (CUPED) that gets the same answer from far fewer users.

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.

The honest headline: this is the only ground-truth method on the list

Most frameworks estimate impact (RICE) or model it (North Star). Microsoft's experimentation platform measures it — a controlled A/B test on real users is the only number that's actually causal. The other methods narrow the candidates; this one decides the winner.

The cost of that rigour: it requires traffic, infra, instrumentation, and a culture that accepts only ~1 in 3 ideas actually win. The famous Kohavi line: "At Microsoft, of every 100 features we ship, only about a third improve the metrics they were designed to improve." Trustworthy experiments are how you find out which third.

The three pieces — ExP, OEC, CUPED

It's tempting to talk about "doing A/B tests" as if it's one thing. Microsoft separates three distinct problems and gives each a name.

ExP

The platform

Internal experimentation system that lets any team design, randomize, deploy, and analyse experiments without re-building plumbing each time. The platform itself is the cultural enabler — without it, experiments are too expensive to run by default.

OEC

Overall Evaluation Criterion

The one agreed success metric per experiment, decided before the test runs. Without an OEC the team will cherry-pick whichever metric moved. The OEC also embeds long-term thinking (e.g. revenue per session weighted by churn risk) so a team can't win by gaming a short-term proxy.

CUPED

Controlled-experiment Using Pre-Experiment Data

A variance-reduction technique (Deng, Xu, Kohavi, Walker, 2013): use each user's pre-experiment behaviour as a covariate to subtract out their baseline noise. Result — same statistical power from ~half the sample size, so experiments read out in days, not weeks.

Source: exp-platform.com (Kohavi's hub) · CUPED paper (Deng et al., 2013) · book: Trustworthy Online Controlled Experiments (Kohavi, Tang, Xu, 2020).

How an ExP-style experiment actually picks the winner

Five steps. The discipline that distinguishes "trustworthy" experiments from "we shipped an A/B test" is that steps 1–2 happen before any code is written.

1

Write the hypothesis and pick the OEC

A single sentence: "If we ship X, we expect the OEC to move by Y, because Z." The OEC is locked before the test starts — no peeking and re-picking.

2

Define guardrail metrics

Metrics that must not get worse: latency, error rate, support volume. Catches the "OEC went up but churn went up too" failure mode.

3

Power-calculate the sample size

How many users for what effect size? CUPED comes in here — by subtracting pre-experiment variance you need ~half as many users for the same confidence.

4

Run the test, then check it ran clean

Sample-Ratio-Mismatch checks, A/A pre-tests, segment analysis. Microsoft's research shows ~70% of "interesting" results were trust-flagged on first inspection — this step is what makes the platform trustworthy.

5

Read the OEC. Ship, kill, or iterate.

If the OEC moved and guardrails held, ship. If not, kill — even if you loved it. The book calls this "humility through data."

Worked example — Bing's $100M headline-color test

Tiny change, enormous OEC movement

The most cited Microsoft experiment: a Bing engineer proposed changing the colour and font of result-page headlines. It sat unprioritised for more than six months — until someone ran the A/B test.

"It produced one of the best results in Bing's history. Increased revenue by approximately $100 million annually." — Kohavi & Thomke, HBR (Sept 2017)

The lesson is not "small changes can be big" — it's that without the experiment, no scoring framework would have ranked this above the work it was beating. The team's pre-test estimate was effectively zero. Only measurement found it.

Source: HBR — The Surprising Power of Online Experiments (Kohavi & Thomke).

Why this matters for the planning debate. If only ~1/3 of ideas win, ranking 50 ideas with RICE and shipping the top 10 means you ship ~7 losers and ~3 winners — and you may not know which were which. ExP-style discipline says: ship behind a test, declare the OEC up front, kill what doesn't move it. That's the entire argument for .
