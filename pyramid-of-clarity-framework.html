Pyramid of Clarity — Asana's mission-to-tasks ladder 2020

Five layers, top to bottom: Mission → Strategy → Objectives → Key Results → Projects (with tasks underneath). Developed internally at Asana and named publicly in their Inside Asana post (Team Asana, August 21, 2020) launching the Goals product.

Asana built it as a lighter alternative to classical OKRs that still ladders every task all the way up to the company mission.

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.
🎓 8th-grade version
Picture a pyramid. At the very top, the company's mission (one sentence — why it exists). Below that, the strategy (how it'll get there). Below that, this period's objectives. Below that, the key results — the numbers that prove the objectives are happening. At the bottom, all the projects (and the tasks inside them) people are actually doing. The rule: every task you work on should connect upward, all the way to the mission. If your task doesn't connect, you've spotted a problem — either fix the connection, or stop doing the task.

The honest headline: alignment by traceability, not by scoring

The Pyramid of Clarity answers one question: "Why are we doing this task?" If you can trace any task upward through Projects → Key Results → Objectives → Strategy → Mission, the work is justified. If the trace breaks anywhere, it's a flag to either reshape the task or — more honestly — drop it.

Like Salesforce's V2MOM, it is not a feature-impact estimator. It tells you whether work aligns, not whether it will succeed. The Pyramid's contribution is the alignment trace itself; the priority signal comes from pairing that trace with measurable Key Results at the middle layer.

The five layers, top to bottom

The shape is literal — wider at the bottom because many tasks roll up to fewer Projects, fewer KRs, fewer Objectives, one Strategy, one Mission. Every layer is supposed to reference the layer above.

MissionWhy the company exists. Decade-scale.

StrategyHow we'll get to the mission. Multi-year.

ObjectivesWhat we want to achieve this period (1–4 quarters).

Key ResultsMeasurable outcomes that say the Objective happened.

Projects (then Tasks)The actual work that moves the KRs.

Asana's framing (verbatim): "Complete a task, and you're making progress on a project. Complete the project, and you're advancing a goal. Advance enough goals, and you're moving toward the mission."

Source: Asana — "Inside Asana" (Team Asana, August 21, 2020). Dustin Moskovitz wrote about related goal-alignment ideas on Asana's Wavelength blog, which has since been retired.

How the Pyramid actually picks work

The mechanism is upward traceability + measurable KRs. Asana built their product so the linkage is enforced — every Project in Asana can be attached to a Goal, which is attached to a parent Goal. The model assumes the structure is visible.

1

The Mission is set and rarely changes

Asana's own Mission: "Help humanity thrive by enabling all teams to work together effortlessly." Decade-scale. Provides the constant North that the rest of the pyramid resolves against.

2

Strategy translates Mission into a few multi-year bets

This is the layer V2MOM's Methods live in. Asana folds it under "Strategy" — same idea, different label.

3

Objectives + Key Results — the OKR layer

Each period (Asana uses quarters), every team writes Objectives that advance one Strategy, with Key Results that are measurable, time-bound, ambitious. This is the only quantitative layer.

4

Projects must be tied to a KR

The forcing function: in the tool, a Project without a parent Goal is visibly unattached. Teams must either tie it to a KR or justify why it's still on the list.

5

Tasks roll up automatically

Daily-level work inherits its justification from the Project it sits in. The promise: any individual can trace why their task matters by walking the pyramid upward.

Pyramid vs V2MOM vs classical OKRs — what's the difference?

These three sit in the same category (alignment, not scoring) and overlap a lot. Quick map so the meeting can pick deliberately.

Pyramid of Clarity (Asana)

  • Five layers: Mission · Strategy · Objectives · Key Results · Projects.
  • Enforced as a tool in Asana — every Project has a parent.
  • Adopts OKR as the measurable middle layer; lighter "hierarchy" than V2MOM.

V2MOM (Salesforce)

  • Five sections per person/team: Vision · Values · Methods · Obstacles · Measures.
  • Each person writes one; cascade up to CEO.
  • Includes Values and Obstacles explicitly — Pyramid does not.

OKRs alone (Doerr / Google) are just Objectives + Key Results, no Mission/Strategy/Values wrapper. The Pyramid and V2MOM both wrap OKRs in additional structure — Pyramid via mandatory parent layers, V2MOM via Values + Obstacles. Choose by what gap your org actually has.

What the Pyramid does not do

It does not score features. A Project tied to a KR is "justified," not "high-impact." If three Projects are all tied to the same KR, the Pyramid doesn't tell you which to do first. That's where RICE, the North Star Framework, or experimentation come in.

It is only as honest as its KRs. If KRs are vanity metrics (output counts, "ship X features"), the Pyramid laddering will look perfect while the company is going sideways. The discipline is forcing outcome KRs, not output KRs.

Apply to a feature sheet

The Pyramid doesn't rank — it enforces upward traceability. If you adopt it, every Project on the backlog must declare its parent KR (which traces to an Objective, then Strategy, then Mission). Orphan Projects — those with no KR parent — are flagged for either re-shaping or dropping. No score, just a trace.

Try it Monday morning (30 minutes). Open your team's current backlog. Add one column: Parent KR. For each item, fill in which Key Result it moves — and be strict. "Engineering hygiene" / "user request" / "the CEO mentioned it" are not KRs. Any row you can't fill in is an orphan. Count the orphans. That number is your honest signal of how much of your roadmap exists because it's on the roadmap, not because it serves a measurable outcome.

For the worked example below, assume this team's pyramid:
Mission: "Help product teams work together effortlessly."   Strategy: "Become the system-of-record for cross-team work."   Objective Q3: "Lift activation & collaboration depth in mid-market."

The most important reading skill on this page. Notice the difference between the "AI insights panel" row and the four In rows above it. All five claim a parent KR. The difference is the Expected contribution column: the In rows state a number ("+4 pp D14 activation"); the AI row says "helps activation — no number." A parent KR you can't quantify against is a vanity trace, not a real one. The Pyramid only filters out the obvious orphans; the harder discipline is forcing a number into the contribution column — because that's where features stop hiding behind their parent goal.
Decision rule. Three gates. (1) Does the Project trace to a KR? Orphans fail the Pyramid — drop or wait for a parent. (2) Is the expected contribution an outcome (a measurable KR move) or just an output (a thing that gets shipped)? Outputs disguised as outcomes are the AI-panel pattern — re-shape until you can state the lift. (3) Once in-scope, rank Projects against each other inside the same KR by Effort / contribution. The Pyramid filters; you still need RICE or experiments to choose between two Projects that both ladder up.
Why the Pyramid is on the list anyway. It is the most concrete artifact of "every task must explain itself." That alone catches the common planning failure mode — items that are on the roadmap because they were on last sprint's roadmap, never asked which outcome they're meant to move.
