Airbnb ERF — standardised analysis as the platform's product 2014

Airbnb's Experiment Reporting Framework (ERF) was built by Will Moss and team when their experiment volume outgrew bespoke analysis. First publicly described in his Airbnb Engineering post on May 29, 2014.

The framework's distinctive contribution: a common metric configuration language and a standardised report per experiment that makes valid analysis the default — not a heroic act per test. Moss's own framing (verbatim): a tool to "make running experiments easier by hiding all the pitfalls and automating the analytical heavy lifting."

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.
🎓 8th-grade version
Airbnb used to test ideas by having a data scientist look at the numbers from each experiment by hand. That works for a few tests — but Airbnb had hundreds running at once. So they built a tool called ERF that does the analysis automatically and prints the same kind of report every time. Two clever things: (1) every team uses the same definition of metrics like "bookings" so no one can argue about whose number is right; (2) the report tells you not just "the number went up" but "the number went up after subtracting the noise" — because sometimes a feature looks like a win but is actually just seasonal traffic the team would have gotten anyway.

The honest headline: standardise the analysis so engineers can run their own experiments

Airbnb's published story (2014–2018 engineering posts) is the painful one familiar to anyone who's tried to scale experimentation: at low volume, you analyse each test by hand. At high volume, the bottleneck becomes the data scientist. ERF was the framework that broke that bottleneck — define metrics once in a config language, and the platform produces a valid analysis automatically.

Same decision rule as the rest of the experimentation set (OEC + guardrails + ramp). What's distinct: variance reduction baked into the platform itself, so smaller tests can detect smaller effects faster — without an analyst's intervention per experiment.

What ERF actually is — three pieces

It's a reporting / analysis layer on top of Airbnb's A/B platform, not a separate testing tool. The pieces are what make experiments cheap to read.

Config

Metrics defined once, in a shared DSL

Each metric (booking rate, search-to-book conversion, host response time…) is described in a single config file. All experiments reference the same definition — eliminating "your numerator vs my numerator" drift.

Variance

Automatic variance reduction (CUPED-style)

Later iterations of ERF applied pre-experiment-data covariates by default to shrink confidence intervals — the same statistical trick that Microsoft's CUPED (2013 paper) uses. The original 2014 ERF post focuses on standardisation; variance-reduction features appeared in later "Scaling ERF" follow-ups.

Reports

Standard reports per experiment

One templated output per test: OEC, supporting metrics, guardrails, significance, confidence interval. Engineers and PMs read it without a data scientist on call.

Sources: Will Moss — "Experiment Reporting Framework" (Airbnb Engineering, May 29, 2014) — the original ERF post · Jonathan Parks — "Scaling ERF" follow-up.

How an Airbnb-style experiment actually picks the winner

Same OEC discipline as everyone else's platform. What changes is who can read the result — and how soon.

1

Engineer / PM defines the experiment in code

Hypothesis, OEC, sample-size estimate. Metrics referenced by name from the shared config — no bespoke calculations.

2

Platform assigns randomization and starts the test

Flag-based deploy. Variance-reduction covariates picked automatically from pre-experiment data for each user.

3

Auto-generated report reads the OEC + guardrails

Same report template every time. Standardisation is the trust mechanism — readers know exactly where to look.

4

Ship, kill, or iterate — author decides

No analyst gate. The author is on the hook for reading the report honestly; the report itself flags caveats (insufficient sample, SRM, etc.).

5

Cumulative metric library improves the next test

Every new metric added to the config becomes available to every future experiment. The framework compounds.

Apply to a feature sheet

ERF doesn't change the columns much — it changes the number you read. The sheet's Result column has two values: raw lift and ERF-adjusted lift (corrected for novelty, seasonality, network effects). Decisions are made on the adjusted number. The classic trap ERF prevents: shipping a feature whose "lift" was actually seasonal demand or a novelty bump.

Try it Monday morning (30 minutes). Pick the last A/B test your team shipped. Find the result. Now answer two questions: was the metric defined in a shared place where another team would use the exact same definition? Did anyone check whether the lift could be explained by seasonality (was there a holiday weekend in the test window)? If either answer is "no," your team is at the pre-ERF stage Airbnb was at in 2013 — and the ERF leverage is in those two specific gaps, not in more sophisticated stats.
Quick glossary for the columns below. Raw lift = the OEC delta straight from the test, as if no adjustments were needed. ERF-adjusted lift = the same delta after accounting for novelty effects (users react to anything new), seasonality (lift may be calendar-driven), and network effects (users in treatment can affect users in control). Novelty effect = the temporary engagement bump that fades after users get used to the change. Seasonality = systematic time-of-week / time-of-year demand patterns that exist independent of the test. The gap between raw and adjusted is the lie ERF prevents you from believing.

The most important reading skill on this page. Compare the "Instant-book promotion" row with the "Wishlist sharing" row. Instant-book: raw +2.0%, adjusted −0.1%. Wishlist: raw +0.4%, adjusted +0.7%. Both gaps are large — but they point in opposite directions. Adjustment doesn't always shrink the lift; sometimes it grows the lift because the raw number was masking a real network effect. This is why ERF is a standardisation tool, not a "make numbers smaller" tool. A team using raw lifts ships the instant-book feature (wrong) and under-ships the wishlist feature (also wrong). Both errors disappear when the analysis is consistent and the report is the same shape every time.
Decision rule. Decide on the ERF-adjusted lift, not the raw. The instant-book row is the canonical trap: raw +2.0% would look like a clean ship, but ERF reveals it as entirely seasonal demand the team would have taken credit for. The wishlist row shows the inverse — ERF can increase lift when a real network effect was being masked. Without standardised reporting, every team interprets their own numbers and trust collapses; with it, the whole company reads the same shape every time.
The transferable insight: standardisation, not statistics. The Airbnb story is sometimes told as a CUPED variance-reduction story. The deeper lesson is that the platform's product is the standardised analysis. Without ERF, every team builds their own analysis and the results aren't comparable. With it, the team can ship 10× more tests because each one costs the same fixed (low) effort to read.
