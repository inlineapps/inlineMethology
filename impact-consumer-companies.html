Study B · Consumer-tech field study

How Top Consumer-Tech Companies Decide What to Build

Ten consumer giants — their product-decision culture, and how they handle a feature's expected impact. The headline: most of them don't estimate impact at all. Every claim is sourced.

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.
Key finding Define impact first Measure, don't estimate All methods compared →

The uncomfortable, grounded finding

Of these ten, only Amazon (PR-FAQ) — and partly Google (HEART/GSM) and Spotify (DIBB) — actually define expected impact before building. The rest — Netflix, Booking.com, Airbnb, Uber, DoorDash, Lyft, Pinterest — are measure-not-estimate cultures: they don't forecast a feature's value, they ship it behind a controlled experiment and let data reveal the truth.

And the truth is humbling: across Microsoft, Google and Bing-scale testing, only ~10–33% of ideas actually move the target metric. That is the real answer to "why don't we know the dead-simple winning feature?" — the best companies assume they don't know and instrument cheaply to find out.

The transferable discipline isn't a magic estimate. It's: define one success metric (OEC) up front, then let data — not the to-do list — decide what ships.

Define Impact Before Building

Force the expected outcome onto paper — or into a metric tree — before writing code.

01Amazon

Define first
Framework: Working Backwards / PR-FAQ  ·  Deep-dive →
  • Most major products since ~2004 begin from the customer experience and work back to what to build.
  • Principal artifact: a mock press release (<1 page) + FAQ (≤5 pages) — a forcing function to define customer value before any code.
  • Narrative 6-page memos, not slide decks; meetings are truth-seeking, not selling.
How they handle impact The clearest pre-build estimation model in the set: the PR-FAQ forces the customer problem, expected outcome, and explicit success metrics to be written down up front. Codified by ex-Amazon execs Bryar & Carr in Working Backwards.
SourceAbout Amazon — culture & processesWorking Backwards — PR-FAQ

02Google

Define first
Framework: HEART + Goals-Signals-Metrics (+ OKRs, experiments)  ·  Deep-dive →
  • HEART — Happiness, Engagement, Adoption, Retention, Task-success — a UX metric framework (Rodden, Hutchinson & Fu).
  • GSM ladders each goal → observable signal → measurable metric, keeping metrics tied to intent.
How they handle impact HEART/GSM defines what success looks like before measuring; actual lift is then validated by experiments. A metric-laddering tool, then test-confirmed.
Original source: Rodden et al., "Measuring the User Experience on a Large Scale," CHI 2010.
SourceIxDF — Google's HEART frameworkGoogle Research — original paper

03Spotify

Define first
Framework: DIBB + Bets Board — Data → Insight → Belief → Bet  ·  Deep-dive →
  • Aligns the org around a consistent path from observed data to committed bets.
  • Company Bets (large, cross-org, 6–12 mo) tracked on a Kanban "Bets Board"; each bet has a 2-page doc with sponsor, stakeholders, and success metrics.
How they handle impact A qualitative/strategic prioritization model: bets are justified by a belief derived from data, with success metrics defined per bet, then validated in execution — not a quantitative estimator.
Sourcing caveat: no first-party Spotify post defines DIBB. Best sources are Henrik Kniberg (ex-Spotify coach) and secondary PM sites — treat as second-party.
SourceHenrik Kniberg — Spotify RhythmProduct Frameworks — DIBB

Measure, Don't Estimate

These companies don't forecast a feature's impact — they ship behind a controlled experiment and read the result. Discipline = a pre-declared success metric + guardrails.  

04Netflix

Measure
Framework: Experimentation & Causal Inference (ABlaze platform)  ·  Deep-dive →
  • Nearly every product/business decision is guided by member behavior observed in tests, not opinion.
  • Where A/B tests are infeasible, uses quasi-experiments and causal ML (interrupted time series, IPW).
How they handle impact Post-hoc measurement, not pre-build estimation — measures causal lift via controlled experiments, using sequential testing and CUPED-style variance reduction to detect effects faster.
SourceNetflix Research — ExperimentationTechBlog — Sequential A/B testing

05Booking.com

Measure
Practice: Democratized large-scale experimentation (OEC + SRM guardrails)  ·  Deep-dive →
  • Any employee can launch an experiment on millions of users without management sign-off.
  • Central platform enforces guardrails: standard randomization, automatic Sample-Ratio-Mismatch checks, pre-defined metrics, hypothesis templates.
How they handle impact Measure-not-estimate: ship decisions rest on the experiment's OEC movement plus guardrails, not pre-launch forecasts. A counterintuitive win they'd never have guessed: showing sold-out properties alongside available ones increased bookings.

06Airbnb

Measure
Framework: Experimentation Reporting Framework (ERF)  ·  Deep-dive →
  • Internal A/B platform for validating hypotheses and quantifying impact; metrics defined in a common config language.
  • Built to make experiments simple and reduce error/bias via automation.
How they handle impact Measure-not-estimate: quantifies impact post-launch via ERF, using variance reduction to shrink confidence intervals and required sample sizes.

07Uber

Measure
Framework: Experimentation Platform (XP)  ·  Deep-dive →
  • Combines staged feature rollout with analytics; flags, monitoring, and a "universal holdout" to measure long-term cumulative effects.
  • Monitors treatment effects to prevent regressions on key metrics.
How they handle impact Measure-not-estimate; impact read from live experiment metrics. Re-architected from remote to local evaluation for 100× lower latency.

08DoorDash

Measure
Framework: Switchback experiments (Dash-AB engine)  ·  Deep-dive →
  • A marketplace breaks standard A/B independence (orders share a Dasher fleet → network effects).
  • Solution: switchback tests randomizing treatment/control by region × time-window, not by user.
  • Dash-AB standardizes A/B, switchback, CUPED/CUPAC and interaction analysis.
How they handle impact Measure-not-estimate, with marketplace-specific design (switchbacks, cluster-robust standard errors) to get unbiased causal estimates despite interference.
SourceDoorDash — Switchback testsMeet Dash-AB

09Lyft

Measure
Practice: Marketplace experimentation (A/B + time-split / region-split)  ·  Deep-dive →
  • Strong norm to test every product change; causal data informs strategic direction.
  • Most common test = rider/driver A/B; second = time-split tests to handle network-effect interference.
  • Investing in adaptive/always-on experimentation (contextual bandits).
How they handle impact Measure-not-estimate; marketplace interference handled via time/region splits rather than naive A/B.
SourceLyft Eng — Marketplace experimentationChallenges in experimentation

10Pinterest

Measure
Practice: A/B testing platform  ·  Deep-dive →
  • Experiments guide products and features; lightweight config UI + QA workflow + simplified cross-platform APIs.
  • Real-time monitoring dashboard to catch regressions early.
How they handle impact Measure-not-estimate; impact read from experiment metrics with real-time monitoring.
