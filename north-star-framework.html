The North Star Framework — and how it actually sets priority 2019

Slack's "2,000 messages" and Amplitude's "Weekly Learning Users" are metrics — not a way to score a feature. This page closes that gap: what the framework is, and the real mechanism by which it decides what to build first.

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.
🎓 8th-grade version

Most teams build whatever ideas sound good. North Star says: pick one number that means "our customers are getting value" (Slack picked teams sending lots of messages). Then list the 3–5 things teams can directly do to move that number — call those the "inputs." Each cycle, pick the weakest input and only fund ideas that touch it; everything else waits its turn.

It's a filter, not a score. Once features are filtered in, you still rank them with RICE or a real experiment.

The honest headline: it is not a scoring formula

If you came expecting a number-per-feature like RICE, there isn't one. The North Star Framework does not score features. It does something earlier and more important: it tells you which single metric you are trying to move right now, and it forces every candidate feature to declare which input it claims to move and by how much.

So "how do Slack/Amplitude decide impact?" — they don't compute an impact score. They model how value is created (North Star = f(a few inputs)), pick the input with the most leverage, and only fund work that moves it. The actual ranking of those candidates still falls to a scoring method (RICE/ICE) or a real experiment. The framework is the filter and the focus, not the scoreboard.

Anatomy: three layers, one causal claim

The framework is a causal model — Cutler and others describe it as a belief map or driver diagram: a written claim about how value is produced. Read it top-down as: "if we move these inputs, this North Star rises, and the business grows."

How the framework actually picks the work

This is the part that's usually skipped. DoubleLoop's blunt line: "most companies completely skip the step of making a model," which is why a North Star metric alone changes nothing. The mechanism is five steps.

1

Build the model

Write the causal map: North Star = f(input₁, input₂, …). Each input is something a team can move directly. This is the step everyone skips — without it the metric is just a poster.

2

Find the leverage point

Ask which input is most movable and most underperforming right now — the one where a small push produces the biggest move on the North Star. That input becomes this cycle's focus, not all of them. (The Playbook frames good strategy as picking where to focus, not what to do — the leverage point is the where.)

3

Set a time-boxed target on it (OKR)

Turn the chosen input into a goal: move this input to this value by this date. The roadmap is now "bets aimed at that one input," not a wishlist.

4

Map each candidate feature to the input it claims to move

For every idea, the question becomes: which input does this move, and by how much? An idea that doesn't touch the focus input is out of scope this cycle — that is the filter doing its job.

5

Ship, then track input and output

Measure whether the input actually moved (did the feature work?) and whether the North Star followed (was the input the right lever?). Both can fail — and that feedback corrects the model for next cycle.

So how do you "score" a feature? You estimate its lift on the input

The framework reframes "impact" from a vague guess about revenue into a concrete, checkable claim about one input. That's far easier to estimate — and far easier to be proven wrong about.

Expected impact of a feature, the North Star way:
impact = (projected lift to the focus input) × (that input's modeled weight on the North Star)
Example: "Onboarding revamp → we believe it lifts % of teams reaching 2,000 messages from 40%→55%; that input is the strongest driver of Weekly Active Teams, so it's high-impact this cycle." Notice the claim is testable, scoped to one input, and time-bound.
The framework does not replace RICE / ICE — it feeds them. The North Star model narrows 50 ideas down to the handful that move the focus input, and gives each a sharper Reach/Impact basis (lift-on-input instead of gut feel). You then rank that short list with a scoring method, or run a fake-door test (a fake button or landing page for a feature that doesn't yet exist, used to measure real interest) or controlled experiment to measure the lift instead of estimating it. Slack vs Amplitude — same framework, different maturity of evidence

Both are real input-modeling cases, but be honest about the sourcing: Amplitude authored the method and documents it; Slack is the famous activation-input story told mostly by its founder, not a published per-feature process.

Slack

North Star · Weekly Active Teams
  • The input that defined focus: a team reaching 2,000 messages sent. Butterfield: "after 2,000 messages, 93% of those customers are still using Slack today."
  • How it set priority: the metric pointed product/onboarding work at getting more teams across that activation threshold faster — integrations, onboarding, invites — rather than scattered feature requests.
  • It's a focus mechanism: the input told them where to aim, not a number per feature.
Caveat: the 2,000-message figure and NSM are founder / secondary accounts (First Round, Amplitude's example), not a published internal scoring sheet.
First Round — Butterfield interview

Amplitude

North Star · Weekly Learning Users
  • Authored the Playbook — the framework's most complete public write-up (NSM + inputs + guardrails + the model).
  • How it sets priority: a "North Star tree" connects each initiative to the input it moves; teams pick bets at the leverage points and set OKRs on inputs.
  • Adds guardrail metrics (churn, support load) so a feature that lifts an input but harms the product is caught.

Apply to a feature sheet

North Star doesn't score features — it filters them. If you adopt it, your backlog gains columns that force every idea to declare which input it claims to move. Features that don't touch this cycle's focus input aren't ranked low — they're out of scope, full stop.

Try it Monday morning (30 minutes). Write your team's North Star at the top of a sheet. List the 3–5 inputs that move it. Pick the one input you'd most want to lift this cycle and circle it. Now go through your last 10 backlog items and write the input each claims to move next to its title. Items that name nothing — or name a different input — are this cycle's "out of scope." That's the framework working. The hardest part is naming the inputs honestly; everything else follows.

The most important reading skill for this sheet. Out · queue ≠ bad idea. Re-engagement, integrations, and search above are all reasonable bets — they just don't touch this cycle's focus input (Depth). They sit in queue, fully credited, until a future cycle's focus shifts to Retention/Breadth/Efficiency. The only rows that get killed outright are the ones that name no input at all (here: Dark mode). That distinction — "wrong cycle" vs "wrong idea" — is what protects the framework from feeling like a kill-list.
Decision rule. Three gates, in order: (1) does the feature name an input it moves? If "none," it's a vanity request — out. (2) Is that input this cycle's focus? If no, queue for a future cycle when that input becomes the focus — not a cut. (3) Is the projected lift plausible — backed by data, analogy, or at minimum a fake-door test plan? Only features passing all three enter the ranked build queue, where you score them with RICE/ICE or run experiments. Illustrative numbers; the framework is the filter, not the scoreboard.
