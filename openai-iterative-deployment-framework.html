OpenAI iterative deployment — staged release as the decision unit 2019

OpenAI has not published a method for ranking which features to build. What it has published — extensively — is how it releases capabilities: in widening rings of scope, with a Preparedness-Framework safety gate between each ring.

The "feature" sheet for this method is really a release queue, not a backlog. Foundational source: the GPT-2 staged release (February → November 2019) plus the August 2019 report "Release Strategies and the Social Impacts of Language Models." Codified later in "Planning for AGI and Beyond" (2023) and the Preparedness Framework (v1 December 2023, v2 April 15, 2025).

Goal
Decide how widely (and how fast) to expose a capability — using real-world signal at each ring, not pre-release estimates.
🎓 8th-grade version
OpenAI doesn't have a famous formula for deciding which features to build. What they do have, in writing, is a careful process for deciding how widely to release a new AI capability. They first show it only to their own researchers. Then to a small outside group. Then to paying subscribers. Then to everyone. Between each step, they check: is anyone misusing this? Is it more dangerous than we thought? If yes, they pause or roll back. They have a separate scorecard — the Preparedness Framework — that checks whether the capability could be used to make bioweapons, do major cyberattacks, or improve AI itself in dangerous ways. The whole idea, first practiced with GPT-2 in 2019: release in small steps so society and the technology can adapt together, instead of dropping the full thing all at once.

The honest headline: this is a release method, not a feature-selection method

If you came here looking for OpenAI's RICE-equivalent or PR-FAQ-equivalent — there isn't one. The chief product officer's published process notes describe quarterly roadmapping, "plans over plan," and strong bottom-up autonomy, but no public scoring formula or kill criteria. What OpenAI has codified, in actual published documents, is the deployment ladder — the stages a capability moves through from internal red-team to broad release, and the safety thresholds that gate each move.

The unit of decision is "this capability, at this scope, with these mitigations" — not "this feature, this quarter, this priority." If you adopt this approach, you're adopting a release discipline, not a prioritisation discipline. Pair it with one of the other methods on this site (Scoring or Define-first) for what to build in the first place.

The scope ladder — five rings, with a gate between each

A capability climbs from narrow internal use to broad release, one ring at a time. Between each ring, a check: has real-world signal at the current ring exposed harm, misuse, or capability surprises that justify pausing or rolling back?

Internal

Researchers + red-team only

The capability is exercised inside OpenAI. Red-team probes for misuse before any external exposure.

Gate to nextPreparedness scorecard run; system card draft started; no Critical-tier risk surfaced.
Research preview

Small external group, gated waitlist

Researchers, alpha partners, hand-picked safety-aware users. The point is real use at low blast radius.

Gate to nextNo new Critical-tier capability discovered in use; misuse rate measurable; mitigations holding.
Limited rollout

Paid tier / beta, with use-case restrictions

Wider but still gated — Plus subscribers, API allowlist, or specific use-cases only. Telemetry now statistically useful.

Gate to nextMisuse-rate stable or declining; system card published; policy review complete.
Broad rollout

Generally available, default-on

Free tier or full API; default-on for new users. Real volume; real harm-surface; full system-card disclosure.

Gate to nextProduction-grade monitoring; rollback path verified; quarterly Preparedness re-evaluation.
Embedded / API-default

Default model behaviour for everyone

Folded into the default model — invisible as a "feature." Continuous Preparedness re-scoring as the capability evolves.

Always-onIf real-world signal escalates to Critical at any stage, roll back one or more rings.
Movement up is irreversible only if nothing escalates — every published release retains a documented rollback path.
"OpenAI's strategy of iterative deployment puts early versions of technology into the world so people can form intuitions and society and the technology can co-evolve."— paraphrase of OpenAI's stated rationale across "Planning for AGI and Beyond" (Feb 2023) and "How we think about safety and alignment". The practice itself was first demonstrated in the GPT-2 staged release (Feb → Nov 2019).

The gate — Preparedness Framework v2 (April 2025)

Between any two rungs, the same scorecard runs: does the capability cross any threshold, in any tracked category? If yes, it does not advance until mitigations are in place. If it scores Critical post-mitigation, it is not deployed at all.

What the Preparedness Framework v2 actually contains

Tracked categories (3)

  • Biological & Chemical capabilities
  • Cybersecurity
  • AI Self-improvement

Tier thresholds (2, simplified from v1's 4)

  • High — could amplify existing pathways to severe harm
  • Critical — could create new severe-harm pathways

Research categories (5, not yet "tracked")

  • Long-range Autonomy
  • Sandbagging (intentional underperformance)
  • Autonomous Replication & Adaptation
  • Undermining Safeguards
  • Nuclear & Radiological

Tracking criteria — a capability must be all 5

  • Plausible
  • Measurable
  • Severe
  • Net new
  • Instantaneous or irremediable

Source: OpenAI Preparedness Framework v2 (PDF, April 15, 2025) · OpenAI announcement. This is OpenAI's only published decision framework — and as you can see, it gates release, not feature choice.

Apply to a feature sheet

The sheet for this method is a release queue — capabilities awaiting promotion to the next rung. Rows aren't ranked by score; they're tracked by current rung, Preparedness status, and whatever real-world signal the current rung is producing.

Try it Monday morning (30 minutes). Pick one feature your team is about to ship. Instead of writing a launch plan, write a scope ladder: which 5% of users would you ship to first? Which 25% next? Which 100% last? For each step, write the one signal that would pause the next promotion. That's a 30-minute version of OpenAI's ladder, stripped of the safety-specific gating. The discipline transfers even when the risk surface isn't bio/cyber/AI-self-improvement — any consumer feature has analogues (complaint rate, refund rate, support ticket spike) that should pause the next rung if they rise.
Quick glossary for the columns below. Rung = the current scope of release (Internal → Preview → Limited → Broad → Embedded). Preparedness tier = the post-mitigation severity score in any tracked category (High or Critical in v2). System card = the public document disclosing red-team findings, model behavior, and known risks for a capability; required to advance past Preview. Misuse rate = real-world signal at the current rung (per-1k-requests or similar) — falling/flat/rising. Rollback path = a verified mechanism to pull the capability back one or more rungs without breaking dependent systems.

The most important reading skill on this page. Look at the "bio-research assistant" row. Verdict: Don't deploy — even though the team built it, even though they could have shipped it internally. The Preparedness tier ("Critical candidate, Bio") means the gate refuses to let it climb the ladder. The transferable insight: most other methods on this site have no equivalent of "don't deploy at all" — RICE ranks low, A/B fails, but the artifact still tends to exist. OpenAI's published method explicitly carves out a verdict that says some capabilities should be built only to confirm they shouldn't be shipped. That's the part smaller teams should copy first: a written list of risk categories that can kill a feature regardless of how well it scored.
Decision rule. Advance only when (a) Preparedness tier is Low/Medium post-mitigation, (b) system card is published or drafted, (c) the rollback path is verified, and (d) real-world misuse rate at the current rung is flat or falling. Hold when any of those is uncertain — re-evaluate in 30 days. Roll back when misuse-rate is rising and mitigations don't dent it. Don't deploy when post-mitigation score is Critical in any tracked category. This is a release rule, not a build rule — see Where it breaks.

Where iterative deployment breaks (be honest about this)

It does not decide what to build. Iterative deployment is silent on which capabilities to develop in the first place. OpenAI hasn't published a feature-ranking method; Kevin Weil (CPO) has described the internal process as "quarterly roadmapping, plans over plan, strong bottom-up" in podcast interviews, but no published framework. You must pair this with a Scoring or Define-first method to fill the upstream gap.

The unit is "capability," not "feature." The ladder is well-defined for things that can be a Preparedness-Framework risk (model capabilities). It bends awkwardly for ordinary product features (a new settings page, a billing change) that have no safety dimension. For those, the gate is non-binding and the method degenerates to "ramped rollout."

It depends on having a safety org of meaningful size. The Preparedness Framework is staffed work: red-teams, evaluation suites, scorecards. Smaller teams can borrow the shape of the ladder (Internal → Preview → Limited → Broad) but not the gating discipline.

Real-world misuse signal is the load-bearing part, and it's noisy. If your monitoring is weak, "misuse rate stable" can hide accumulating harm. The ladder only works as well as your monitoring at each rung — much like Uber's universal holdout is only honest if the metric is honest.
Why it's on the list. The user asked specifically whether AI labs have published a method for deciding what to ship. OpenAI's honest answer is "not for choosing, but yes for releasing" — and the Preparedness Framework + iterative-deployment posts are the most-cited publicly published decision discipline either lab has produced. Worth knowing, even if it only fills half the loop.
