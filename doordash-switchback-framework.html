DoorDash switchback experiments — measuring causally on a shared marketplace 2018

In a marketplace, users share resources (Dashers, restaurants, supply). A normal A/B test gets contaminated: changing one user's experience affects others.

DoorDash's published answer — switchback testing on region × time blocks — gives unbiased causal estimates despite the interference. First publicly described in the DoorDash Engineering blog post of February 13, 2018.

Goal
Decide features by data-backed expected impact — choose by outcome, not by to-do list or opinion.
🎓 8th-grade version
Normally an A/B test gives half of users the new thing and half the old thing, then compares. But on DoorDash, the half with a "new pricing" idea pulls Dashers away from the other half — so the "old" half is also affected by the new pricing, and the test is lying about how much the new thing actually helped. DoorDash's trick: switch the whole region on for 30 minutes, then off for 30 minutes, then on again, etc. — so everyone in the region has the same experience at any given moment, and you can compare "on" windows to "off" windows. This is a switchback test, and it's how you get honest answers when everyone shares the same pool of resources.

The honest headline: a marketplace breaks ordinary A/B tests

If you randomly assign half of users to a new pricing algorithm in a shared Dasher pool, the new algorithm changes Dasher availability for the control users too. The control group is no longer truly "without the change" — they're affected by it through the shared resource. The measured lift is biased downward (often by a lot).

DoorDash's contribution: switch the whole region's algorithm on/off on a rotating schedule, then compare the two states within the same region. The shared resource is now part of the unit being tested. This is the standard playbook for marketplace experiments (Lyft and Uber publish related variants) and DoorDash's Dash-AB engine standardises it.

The interference problem in one paragraph

Why naive A/B fails on a marketplace

Imagine you ship a new "Surge pricing" algorithm to 50% of customers in San Francisco. The other 50% still see the old algorithm. But all 100% of customers compete for the same Dashers. The treatment group's higher prices pull Dashers away from control-group orders, making control orders slower — which makes control look worse, inflating treatment's measured lift. Or vice versa. Either way, the A/B result is biased and you don't know by how much.

The technical name: SUTVA violation (Stable Unit Treatment Value Assumption). It applies to any marketplace, two-sided platform, or shared-resource feature.

What a switchback experiment actually looks like

Instead of randomising users, randomise region × time window. The whole region runs the new algorithm for 30 minutes, then the old algorithm for 30 minutes, then new, etc. Compare across windows.

How DoorDash's switchback decisions get made

1

Diagnose: is this a marketplace-shared feature?

If the feature changes the shared pool (Dasher allocation, pricing, dispatch) → marketplace test. If not (UI, copy, button colour) → ordinary user-level A/B. This routing decision is the first thing.

2

Pick the unit: region × time window

Choose grid granularity. Too small → not enough Dashers per cell, noisy. Too coarse → few cells, low statistical power. DoorDash's choice depends on the metric's natural cadence.

3

Randomise treatment/control across cells

Dash-AB handles the assignment; the platform ensures balanced regions and times so no cell systematically gets the morning rush.

4

Run, then analyse with cluster-robust SE

Cells inside the same region/time aren't independent — observations cluster. Dash-AB uses cluster-robust standard errors so significance isn't overstated.

5

Read the OEC. Ship, kill, or iterate.

Same decision rule as ordinary A/B — the difference is that the lift estimate is unbiased for marketplace effects.

Apply to a feature sheet

Switchback adds two DoorDash-specific columns to the standard experiment ledger: marketplace flag (does this change a shared resource? routes the design) and switchback grid (region × time-window cells, with cluster-robust SE). User-level features go through ordinary A/B; marketplace-level features must use switchback or the lift estimate is biased.

Try it Monday morning (30 minutes). List your team's last 5 shipped experiments. For each one, ask: could a user in the treatment group affect a user in the control group through any shared resource — supply, ranking, inventory, support queue, social graph, ads inventory? If yes for any, your A/B test was likely biased — and you don't know in which direction. You don't need to immediately switch to switchback; the first step is just tagging these experiments so future readouts mark the bias risk on the spreadsheet.
Quick glossary for the columns below. SUTVA = Stable Unit Treatment Value Assumption — the textbook A/B requirement that one user's outcome can't depend on another user's treatment. Marketplaces, social graphs, and shared inventories routinely break this. Switchback = randomization at the (region × time-window) level rather than the user level, so the shared resource is part of the unit being tested. Cluster-robust SE = a standard-error correction that accounts for observations inside a cell not being independent of each other (DoorDash uses these because lots of orders happen in one region-window). Cell = one (region × time-window) combination.

The most important reading skill on this page. Look at the "Marketplace?" column. Of the 8 features, 7 are "Yes" and one is "No" (the AI ETA promise — pure UI display). The "No" row uses ordinary A/B and works fine. The 7 "Yes" rows all require switchback — and for each, the alternative wasn't "do A/B with a footnote about bias"; it was "use the wrong design and silently believe the wrong number." The transferable insight isn't switchback as a technique; it's the routing decision in the first column. A team that doesn't ask "does this share a resource?" routinely mis-measures features for years and never knows.
Decision rule. The first decision is which design, not which feature. Anything that changes a shared resource (dispatch, surge, Dasher allocation, batching) must use switchback or the lift estimate is biased — likely by enough to flip ship/kill. The aggressive-batching row shows the dual-OEC pattern (delivery throughput up, customer delay up — the second guardrail kills the win). The two-zone pilot's "Extend" verdict is the under-powered case: not a fail, just not enough cells yet.
The transferable insight: design before stats. The wrong analysis can't rescue the wrong experimental design. The DoorDash lesson is to diagnose interference first — most teams don't, and quietly mis-measure marketplace features for years. Even at small scale, asking "does this change a shared resource between users?" is free, and it changes the design.
